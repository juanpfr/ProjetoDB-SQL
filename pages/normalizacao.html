Normalização das Tabelas

Primeira Forma Normal (1FN)

Uma tabela está na primeira forma normal quando:

  1. Existe uma chave primária.
  2. Somente possui valores atômicos (cada célula contém um valor indivisível).
  3. Relação não possui atributos multivalorados ou aninhados.
  4. Relação não possui atributos compostos.

Exemplo de valores atômicos:

  • Não atômico: "Marta Santos" (pode ser dividido em partes significativas)
  • Atômico: "Marta" (nome próprio), "Santos" (sobrenome) separados

Exemplo de atributos compostos:

  • Endereço
  • Rua, Número, Bairro, Cidade, Estado, CEP

Avaliação das Tabelas - 1FN

  • tbl_aluno - Normalizada
  • tbl_historico - Normalizada
  • tbl_professor - Normalizada
  • tbl_matricula - Normalizada
  • tbl_curso - Normalizada
  • tbl_departamento - Normalizada
  • tbl_turma - Normalizada
  • tbl_disciplina - Normalizada
  • tbl_curso_disciplina - Normalizada
  • tbl_professor_disciplina - Normalizada

Segunda Forma Normal (2FN)

Uma tabela está na segunda forma normal quando:

  1. Está na 1FN.
  2. Todos os atributos não-chave são funcionalmente dependentes da chave primária completa.
  3. Não existem dependências parciais.
  4. Se existir dependência parcial, deve-se criar uma nova tabela com esses atributos.
  5. Um atributo-chave é uma PK ou parte de uma PK composta.

Exemplo de violação da 2FN:

RA_Aluno Nome_Aluno Código_Disciplina Nome_Disciplina Nota
555 Juliana BD01 Banco de dados 8.5
555 Juliana PT01 Programação Python 7.0

Neste caso:

  • Chave composta: (RA_Aluno, Código_Disciplina)
  • Nome_Aluno depende apenas do RA_Aluno
  • Nome_Disciplina depende apenas do Código_Disciplina

Solução:

  • Aluno (RA_Aluno, Nome_Aluno)
  • Disciplina (Código_Disciplina, Nome_Disciplina)
  • Notas (RA_Aluno, Código_Disciplina, Nota)

Terceira Forma Normal (3FN)

Uma tabela está na terceira forma normal quando:

  1. Está na 2FN.
  2. Não há dependência transitiva entre atributos não-chave e a chave primária.
  3. Todo atributo não-chave depende diretamente da chave primária.
